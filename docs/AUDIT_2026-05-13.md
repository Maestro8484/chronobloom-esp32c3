## 2. Maturation Goal Summary

> Target: flashable, debuggable, and verifiable entirely over WiFi with the clock mounted on the wall.

All recovery and configuration paths are now available without physical access:
- OTA firmware update: via `http://esp32c3-v3-8inch.local/update`
- WiFi credential update: via `http://esp32c3-v3-8inch.local/wifi`
- Factory reset: hold UP+DOWN at power-on for 3s → portal on `esp32c3-clock-setup`
- Diagnostics: `GET http://esp32c3-v3-8inch.local/diag`
- Watchdog: 10s window auto-reboots device if loop stalls

The only remaining USB dependency is first-time flashing from scratch.

## 3. Documentation Consistency Issues

### 3a. RING_PIXEL_OFFSET — CHANGELOG vs Code Mismatch (IMPORTANT)
**CHANGELOG [2.0.8]** states: `RING_PIXEL_OFFSET 8inch: 1 → 2`  
**Current `platformio.ini`** shows: `RING_PIXEL_OFFSET=1`

Either the changelog entry is incorrect (the actual change was not 1→2 but something else), or a subsequent rollback to 1 is not documented. The REVIEW.md Issue A fix description says `0 → 1`, consistent with the current value of 1.

**Action required**: Verify correct value by physical test (does 12 o'clock align correctly?) and update CHANGELOG [2.0.8] or platformio.ini to match reality.

### 3b. EEPROM Settings Version — FEATURES.md Stale
**FEATURES.md** `Settings Structure (EEPROM v8)` section still shows version 8 as current.  
**REVIEW.md** Issue A fix description mentions settings version bumped `9 → 10` for the sacrificial pixel offset fix.  
**Action required**: Read `SETTINGS_VERSION` from `src/main.cpp` and update the FEATURES.md version history and struct definition to match. Also add `outerRingOffset` (uint8_t) to the struct — it was added in REVIEW.md Issue B but is absent from the FEATURES.md schema.

### 3c. README.md — Stale OTA Command Reference
**README.md** line 84: "Uses IP `192.168.1.110` (set in platformio.ini). Update if the device IP changes."  
**Current `platformio.ini`**: 8inch env uses `upload_protocol = esptool` / `upload_port = COM6` — no espota, no static IP.  
This line is left over from when espota was the default (pre-v2.0.8). It no longer applies.  
**Action required**: Update the "PlatformIO / VS Code upload button" OTA section in README.md to reflect esptool/COM6 as current default, or note that OTA is preferred via `/update` web page and reserve esptool for recovery.

### 3d. FEATURES.md — Diagnostic Endpoint Still Listed as Planned
**FEATURES.md** Medium Priority planned features includes:
```
- [ ] **Diagnostic endpoint** — `/debug` with uptime, heap, NTP sync count, button press counters
```
This was implemented as `/diag` in v2.0.6. Should be removed from Planned or moved to Implemented.

### 3e. FEATURES.md — Physical Buttons Section Incomplete
The Physical Buttons feature block is accurate for v2.0.7 behavior. No issues. GPIO5=UP, GPIO9=DOWN is consistent across FEATURES.md, CHANGELOG, platformio.ini, and README.

### 3f. SESSIONS.md — Session 9 Has No Completed Detail Block
All other completed sessions (2, 4, 5, 6, 7) have a narrative block under "Completed Sessions". Session 9 (WiFi UI, v2.0.9) only appears in the status table as Done. Low priority but worth adding for consistency.

### 3g. ARCHITECTURE.md — FocusReminder EEPROM Field Naming
ARCHITECTURE.md shows `focusReminder_enabled`, `focusReminder_startHour`, etc. (with `Reminder_` infix).  
FEATURES.md shows `focusEnabled`, `focusStartHour`, etc. (without infix).  
One of these must be wrong. The actual struct in `src/main.cpp` is the authority.  
**Action required**: Verify struct field names in `src/main.cpp` and fix whichever doc is wrong.

### 3h. CHANGELOG [2.0.2] vs [2.0.8] — Sacrificial LED Lifecycle Gap
- [2.0.2] says "Sacrificial LED removed" (RING_PIXEL_OFFSET set to 0, SACRIFICIAL_PIXEL_ENABLED=0)
- Between v2.0.2 and v2.0.8, the user physically re-added the sacrificial LED
- [2.0.8] sets SACRIFICIAL_PIXEL_ENABLED=1, RING_PIXEL_OFFSET from 1→2 (per changelog), but there is no changelog entry clearly stating "sacrificial LED re-added" and explaining why
- REVIEW.md Issue A (dated 2026-05-09, before v2.0.6) documents the re-add and its rationale

**Action required**: Add a brief note to CHANGELOG between v2.0.2 and v2.0.5 (or to REVIEW.md) documenting that the sacrificial LED was physically re-installed, why (level-shifting for data integrity), and which config change corresponds.

## 4. Unresolved Technical Issues

These are known open issues documented in REVIEW.md but not yet addressed in code:

| Issue | File | Priority | Notes |
|-------|------|----------|-------|
| Compositing model fragility (§6) | `ClockRenderer::render()` | Medium | 4-pass render with implicit pixel priority; a future feature (e.g. progress arc past minute position) could silently break display. Recommended fix: explicit 60-element composite array before write. |
| Wi-Fi interrupt jitter on LEDs (§3) | `loop()` / `strip_.show()` | Low | FreeRTOS Wi-Fi task can preempt `strip_.show()` on single-core ESP32-C3. Visible as occasional LED flicker. No fix currently; disabling center idle animation would confirm source. |
| `setBrightness()` rounding accumulation (§2/§5) | `LuxSensor::autoBrightnessCached()` | Low | Only relevant if smooth brightness fading is added. Current binary day/night is safe. |
| FocusReminder animation overlap (ARCHITECTURE.md) | `FocusReminderScheduler` | Low | Reminder animations can overlap with status/chime animations. No queue in v1; noted as v2.1 work. |
| FocusReminder test-now button | WebUI | Low | Manual test requires changing time to window; no one-click trigger. Listed in FEATURES.md Planned. |
